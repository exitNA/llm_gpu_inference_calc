# 技能自动发现与沉淀

## 目标

- 识别对后续任务有长期价值的规则，而非只解决一次性请求。
- 优先做“删减和收口”，避免 skill 再次膨胀。
- 默认优先更新现有 skill，不新增同义变体。

## 自动触发策略

1. 在每次任务开始时先做一次轻量扫描，判断是否存在可沉淀规则。
2. 若命中“识别信号”且通过“价值评估”，立即执行沉淀流程。
3. 若未命中，不打断当前任务，不额外输出沉淀信息。
4. 若用户明确要求“不要沉淀/不要更新 skills”，本轮跳过自动触发。

## 识别信号

- 从以下输入提取候选规则：
1. 用户在当前请求明确提出的原则、约束、偏好、流程。
2. 最近多轮对话中重复出现的同类要求。
3. 可跨任务复用的操作顺序、质量门禁、交付格式。
- 排除以下内容：
1. 一次性上下文（临时路径、单次异常值、仅当前分支有效的细节）。
2. 与现有 `docs/*`、代码实现或仓库结构冲突的指令。
3. 无法稳定复用的个人化口头偏好。

## 价值评估（必须）

- 对每条候选规则评估 4 项（是/否）：
1. 普适性：是否适用于同类任务，而非仅当前请求。
2. 稳定性：未来较长时间内是否仍然成立。
3. 可执行性：是否可以转化为明确步骤或门禁。
4. 收益性：沉淀后是否明显减少重复沟通或返工。
- 满足至少 3 项才进入沉淀流程。

## 路由策略

1. 检索现有 skills：优先在 `.agents/skills/*/SKILL.md` 中查找同主题技能。
2. 判断承载关系：
- 若现有 skill 已覆盖同一领域，更新该 skill。
- 若候选规则跨多个现有 skill，更新最贴近的主 skill，并在内容中加交叉引用。
 - 仅当无任何 skill 可合理承载时，创建新 skill。
3. 避免重复创建：同义但边界重叠的技能一律合并，不拆分“同一能力的多个变体 skill”。

## 更新现有 skill 的执行要求

1. 保持原 skill 定位不变，仅补充可复用规则、门禁、检查清单或输出模板。
2. 不引入与现有规则冲突的条款；若冲突，先统一口径再写入。
3. 对新增规则给出触发条件，避免“全局泛化误触发”。
4. 保持精炼：优先删除重复、过期、仓库无对应文件的内容。

## 创建新 skill 的执行要求（仅在必要时）

1. 使用 `skill-creator` 的初始化脚本创建骨架。
2. frontmatter 仅保留 `name` 与 `description`。
3. 在 description 明确“做什么 + 何时触发”。
4. 优先写流程与门禁，避免冗长背景说明。
5. 如无必要，不创建 `scripts/`、`references/`、`assets/` 空目录。

## 标准流程

1. 提取候选规则并完成价值评估。
2. 在现有 skills 中检索可承载位置。
3. 选择“更新”或“新建”，默认更新。
4. 执行文件改动并保持结构简洁。
5. 优先做本地一致性检查；若仓库没有 `quick_validate.py`，手动检查 frontmatter、引用路径和 `agents/openai.yaml`。
6. 输出沉淀结果：规则摘要、落盘位置、为何更新/新建、后续触发方式。

## 交付格式

- 使用以下结构汇报：
1. `Candidate Rules`: 提炼出的候选规则（1-5 条）。
2. `Decision`: `update-existing` 或 `create-new`，并说明理由。
3. `Changes`: 修改的 skill 文件路径。
4. `Validation`: 校验命令与结果。
5. `Adoption`: 后续在什么请求下会自动触发该沉淀内容。

## 硬性限制

- 不将用户隐私、敏感数据或一次性业务事实写入 skill。
- 不把临时 workaround 沉淀为长期规则。
- 不在未评估现有技能承载能力前直接新建 skill。
- 不生成与项目既有工程规范冲突的规则。
