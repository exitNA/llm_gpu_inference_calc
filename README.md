# GPU Analysis

一个基于 Gradio 的单模型推理 GPU 资源估算器，用于按文档《`docs/llm_gpu_inference_calc.md`》的方法评估在线推理的显存下界、吞吐必要条件下界和时延必要条件。
当前交互版以“模型预设 + GPU 规格预设 + 峰值 QPS 或日均调用量 + P95 长度 + TTFT/E2E 目标”作为主要输入，适合做采购前容量规划和能力回推。

## 运行

```bash
uv sync
uv run python main.py
```

默认启动 Gradio 页面，监听 `0.0.0.0:7860`。可选参数：

```bash
uv run python main.py --host 127.0.0.1 --port 7860
uv run python main.py --dry-run
```

`--dry-run` 会使用内置示例参数执行一次估算并直接输出文本结果，不启动 Web UI。

## 交互输入

- 模型参数：按模型系列和版本选择预设
- GPU 规格：按 GPU 系列和版本选择预设，并设置精度、`η_vram`、`η_bw`、`η_cmp`
  - 推理精度支持 `INT4 / INT8 / FP8 / FP16 / BF16`
  - `KV cache` 精度是独立输入参数，默认 `FP16`，用户可自行切换为 `INT4 / INT8 / FP8 / FP16 / BF16`
  - 若显卡预设未提供独立 `INT4 TFLOPS`，当前实现会暂用 `INT8` 算力口径做 `INT4` 理论估算，便于先做容量比较
- 业务目标：
  - `QPS 建模方式`：直接输入峰值 QPS，或按“日调用量 + 高峰放大系数 + Poisson 时间窗分位数”反推峰值 QPS
  - `在途建模方式`：按 `λ_peak × E2E_p95 × 安全系数` 的 Little 近似，或直接输入峰值在途请求量
  - `P95` 输入输出长度、`TTFT/E2E` 目标（单位秒）
- 显存附加系数：`α_w`、`α_r`

## 单位约定

- 显存容量配置使用 `GB`
- 显存展示统一使用 `GB`
- 显存带宽配置使用 `GB/s`
- 模型精度、KV cache 等底层大小使用 `bytes`

## 默认业务画像

- 默认画像继续使用“直接输入峰值 QPS + Little 定律近似在途”的兼容模式
- 默认时延目标以秒级在线问答场景为例
- 峰值在途预算额外乘以 `1.1` 安全系数

## 输出结果

- 决策摘要：业务基线卡数 `G_req`、主导约束、QPS/在途采用口径、时延风险、估算总成本
- 计算图：放在结果区最后，以 DAG 形式展示“输入口径 → 中间指标 → 约束判断 → 最终结果”的依赖链，方便观察某个输入变化后会联动哪些指标
- 约束拆解：`G_mem`、`G_pre`、`G_dec`、两条时延必要条件；吞吐卡片内会直接展示关键公式与本次输入代入值，并明确标注为理论必要条件口径
- 近似能力回推：基于必要条件模型线性回推的总 Prefill / Decode 理论吞吐、理论可持续 QPS、理论每日 token 总量，以及显存口径最大在途请求量
- 业务与 Cache 明细
- 计算过程明细：展示直接峰值 / Poisson QPS / 直接在途三种口径下实际采用的公式、代入值和结果，便于人工验证
- 原始 JSON 结果，便于继续集成或导出

其中当前版本的 `G_req` 先按显存约束 `G_mem` 给出，`G_pre / G_dec` 仅作为吞吐必要条件参考，不再直接抬高采购卡数。“每日 token 供给上限”和“可持续 QPS”都按这组显存基线卡数做理论近似回推；没有实测 profile 时，吞吐相关结果不应直接当作线上稳定吞吐或 SLA 承诺。
